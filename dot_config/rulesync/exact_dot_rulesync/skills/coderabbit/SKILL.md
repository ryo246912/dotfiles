# CodeRabbit CLI によるコード自動レビュー

## 概要

CodeRabbit CLI を使用して、プルリクエストやコミットに対して自動的にコードレビューを実行します。レビュー結果は、プロンプトモードまたはプレーンモードで取得できます。

## 役割と作業フロー

### 1. モードの選択

CodeRabbit は 2 つのモードで実行できます。

#### プロンプトモード（デフォルト）

```bash
coderabbit --prompt-only
```

- CodeRabbit の対話的なプロンプトを通じてコードレビューを実行
- ユーザーとのやり取りを通じて、詳細なレビューコメントを生成
- PIR（Pull Request Review）の形式で結果が得られる

#### プレーンモード

```bash
coderabbit --plain
```

- シンプルなテキスト形式でコードレビュー結果を出力
- プログラム的な処理や自動化に適した形式
- 構造化されたレビュー内容を取得できる

### 2. 実行方法

重要: CodeRabbit は必ずバックグラウンドタスクで実行してください。他のコマンドを同時に実行しないでください。

```bash
coderabbit [--prompt-only | --plain]
```

### 3. レビュー結果の活用

- CodeRabbit からのレビューコメントを確認
- レビュー内容に基づいてコードを改善

## オプション設定

CodeRabbit の実行時に、以下のオプションを指定できます。

- `--prompt-only`: 対話的なプロンプトモードを使用（デフォルト）
- `--plain`: シンプルなテキスト形式でレビュー結果を出力
- その他のオプションについては、`coderabbit --help` で確認

## 4. コミット不要のローカル自動修正ループ

CodeRabbit CLI はワーキングツリーの差分（未コミット変更）を直接レビューできるため、コミットや PR 作成をせずに「レビュー→修正→再レビュー」を繰り返すループが組めます。GitHub 上の PR レビューとは独立した、コミット前のセルフレビューループとして使ってください。

Claude Code / Codex CLI では、この 1 回分（指摘取得→ブロック）が Stop hook (`coderabbit-review-hook.sh`) により自動化されている。実装が完了してエージェントが応答を終えようとするたびに未コミット差分を `coderabbit --plain` でレビューし、指摘があれば stop をブロックして差し戻す。以下の手順は、hook 経由で差し戻された後にエージェントが従う流れ、またはコミット前に手動でループを回したいときの手順として使う。

手順:

1. `coderabbit --plain` を実行し、現在のワーキングツリー差分に対する指摘を取得する
2. 指摘がなければループ終了
3. 指摘があれば、指摘内容に基づいてコードを修正する（このステップでコミットはしない）
4. 再度 `coderabbit --plain` を実行し、指摘が解消されたか確認する
5. 指摘がなくなるか、意味のある改善が見られなくなるまで 2〜4 を繰り返す（目安: 最大 5 回程度で頭打ちなら、残りの指摘を要約してユーザーに判断を委ねる）

注意点:

- 無限ループを避けるため、繰り返し回数の上限を必ず設ける
- 同じ指摘が形を変えて繰り返される場合は機械的に修正を続けず、指摘の妥当性をユーザーに確認する
- このループが完了してから通常のコミット・push を行う

## 5. PR ベースの自動レビューループ（コミット後）

`.coderabbit.yaml` の `reviews.auto_review.enabled` / `auto_incremental_review` が有効な場合、push するたびに CodeRabbit の GitHub App が PR 上へ自動でレビューを投稿する。これを使ってコミット後のループを組む場合の手順:

1. 変更をコミットして push する（PR が無ければ作成する）
2. CodeRabbit（`coderabbitai[bot]`）からの新しいレビューコメントを取得する
   - `gh pr view <PR番号> --json comments,reviews` は会話コメントとレビューサマリーしか含まないため、ファイル行に紐づくインラインコメントは `gh api repos/<owner>/<repo>/pulls/<PR番号>/comments --paginate` で別途取得する
   - 直近の push 時刻より後に投稿されたレビューかどうかを確認し、まだ投稿されていなければ少し待ってから再確認する
3. 指摘がなければループ終了
4. 指摘があれば、ステップ 4（ローカルループ）と同様に修正し、再度コミット・push する
5. `auto_incremental_review` により再レビューが投稿されるので、指摘が解消されるか意味のある改善が見られなくなるまで 2〜4 を繰り返す（繰り返し回数の上限を必ず設ける）

注意点:

- push は共有履歴に影響する操作のため、実行前にユーザーへの確認方針（都度確認 or 一定範囲で自動化を許可）に従うこと
- Claude Code on the web など webhook 経由で PR イベントを受け取れる環境では、ポーリングの代わりにその通知機構を使う方が効率的
- ローカルループ（4.）で指摘を出し切ってから push すると、PR ループの往復回数を減らせる
