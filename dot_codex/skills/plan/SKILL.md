# 実装計画の作成

## 概要

エージェントを用いて実装計画を立て、`plan` ディレクトリにマークダウンファイルとして保存します。

## 実装計画フローとは

実装計画フローとは、以下のフェーズからなる計画手法です。

### 1. 事前準備フェーズ

- ユーザーがエージェントに対して、実装したいタスクの概要を伝える
- `./plan` ディレクトリが存在しない場合は作成する
  - 実行コマンド: `mkdir -p ./plan`
- `./plan` 内にタスクの概要から適切なファイル名を考えて、マークダウンファイルを作成します
  - たとえば、「ユーザー認証機能を追加する」というタスクなら `./plan/add-user-authentication.md` という名前のファイルを作成します
  - ファイル名は英語のケバブケース形式で命名します

### 2. 要件分析フェーズ

- エージェントがユーザーから伝えられたタスクの概要を分析する
- 以下の観点で要件を整理する
  - タスクの目的・背景
  - 満たすべき機能要件
  - 考慮すべき非機能要件（パフォーマンス、セキュリティ、保守性など）
  - 制約条件（技術スタック、既存コードとの整合性など）
- これらを「要件セクション」として計画ファイルに記載する

### 3. 実装計画作成フェーズ

- エージェントが要件分析に基づいて、具体的な実装計画を作成する
- 実装計画には以下を含める
  - 実装する機能・変更の一覧
  - 各実装項目の詳細な手順
  - 実装の優先順位
  - 想定される技術的課題とその対応策
  - テスト計画
  - デプロイ・リリース計画（必要な場合）

### 4. レビューフェーズ

- エージェントがユーザーに対して「実装計画ファイル」を提示し、問題がないかを尋ねる
- ユーザーが「実装計画ファイル」を確認し、問題があればエージェントに対してフィードバックする
- ユーザーが「実装計画ファイル」を確認し、問題がないと答えるまで「実装計画ファイル」に対して修正を繰り返す

### 5. 実装フェーズ（オプション）

- ユーザーが承認した場合、エージェントは「実装計画ファイル」に基づいて実装を開始する
- 実装時は計画ファイルに記載された要件・手順を守りながら進める
- 実装中に計画から逸脱する必要が生じた場合は、ユーザーに確認する

## 実装計画ファイルのフォーマット

実装計画ファイルは以下の構造で作成します。

```markdown
# [タスク名]

## 概要

- タスクの目的と背景を簡潔に記述

## 要件

### 機能要件

- 実装すべき機能の一覧

### 非機能要件

- パフォーマンス、セキュリティ、保守性などの要件

### 制約条件

- 技術スタック、既存コードとの整合性などの制約

## 実装計画

### 1. [実装項目1]

- 詳細な手順
- 影響範囲
- 技術的考慮事項

### 2. [実装項目2]

- 詳細な手順
- 影響範囲
- 技術的考慮事項

## 技術的課題と対応策

- 想定される課題とその解決方法

## テスト計画

- 単体テスト
- 統合テスト
- E2Eテスト（必要な場合）

## デプロイ・リリース計画

- デプロイ手順
- ロールバック計画
- リリース後の監視項目

## タイムライン（オプション）

- 各フェーズの想定期間

## 参考資料

- 関連するドキュメント、技術記事、既存コードへのリンク
```

## 注意事項

- 計画ファイルは `./plan` ディレクトリに保存します
- ファイル名は英語のケバブケース形式で命名します
- 実装計画は具体的かつ実行可能な内容にします
- 既存のコードベースやプロジェクトの慣例を考慮に入れます
- ユーザーの承認を得てから実装フェーズに進みます
